# 影片详情工作台设计

日期：2026-05-25

## 背景

当前影片详情页的主要问题不是缺少后端能力，而是前端把整理动作拆散和隐藏了：

- 页面主视觉仍然偏“展示页”，大量区域只是摘要或统计。
- 观看状态、评分、评论等核心整理动作需要额外进入编辑态。
- 封面、截图、标签、类型、演员、文件等能力在代码中已有对应组件或实现，但没有被组织成统一的主工作流。
- 新建影片进入详情页后，首屏几乎全是空摘要，缺少明确的“下一步整理入口”。

用户目标非常明确：进入影片详情页后，应该能直接在单页内完成整理，而不是只能先看，再想办法找到编辑入口。

## 目标

1. 把影片详情页从“展示页”改成“单页整理工作台”。
2. 第一屏直接提供最高频整理动作：状态、评分、标题、封面、截图、快速关联。
3. 标签、类型、演员、文件、评论、备注、来源信息都在当前页完成编辑。
4. 新建影片进入详情页时，不再出现“空而无用”的摘要感，而是立即进入可整理状态。
5. 尽量复用现有的元数据、关系整理、媒体资产等组件能力，避免重复造轮子。

## 非目标

1. 不把影片详情页拆成多标签页或多个独立编辑弹窗。
2. 不把整理动作迁移到右侧抽屉或二级页面。
3. 不在本期重做整个数据模型或后端命令体系。
4. 不在本期实现复杂协作、版本历史或自动保存冲突解决机制。

## 方案选择

采用“方案 2 + 顶部快编”的单页工作台方案。

核心思路：

- 第一屏先解决最高频动作。
- 页面中段按工作区组织整理模块。
- 所有主要编辑动作都保留在当前页完成。

与“纯展示详情 + 编辑抽屉”相比，本方案更符合用户当前工作方式。与“完全信息密度拉满的单页工作台”相比，本方案保留了更清晰的结构分区，更适合当前项目已有代码基础。

## 信息架构

### 第一屏：顶部快编区

第一屏必须直接可操作，而不是只读展示。

建议包含以下内容：

#### 1. 左侧媒体主卡

- 当前封面预览
- 上传封面
- 粘贴封面
- 添加截图
- 刮削图片或刮削信息入口

目标：用户进入页面后，不需要先找到别的区域，就能立刻补齐最显眼的媒体资产。

#### 2. 右侧基础信息与主操作区

- 番号
- 系列
- 当前整理状态标签，例如“新建未完善”
- 标题 / 日文标题
- 发行日期
- 时长
- 来源站点 / 来源 URL
- 保存改动
- 刮削信息
- 添加文件
- 删除影片

这里不应继续保留“只有展示，没有编辑”的状态。基础元数据应该支持直接编辑或处于明显可编辑结构中。

#### 3. 快编动作条

第一屏紧接基础信息后，直接放出以下快编动作：

- 观看状态切换
- 评分设置
- 快速添加标签
- 快速添加类型
- 快速添加演员

原因：

- 这是整理过程中最常用、最应该短路径完成的动作。
- 现有页面把这些能力藏得过深，导致首屏价值极低。

## 第二层：中段工作区

第一屏之下按工作类型分为四个主区块。

### 1. 关系整理区

负责：

- 标签增删
- 类型增删
- 演员增删

设计要求：

- 当前已关联对象以标签/胶囊形式展示
- 每个对象都能直接移除
- 区块底部提供明确的添加入口，不跳别页
- 演员建议支持搜索后直接关联
- 标签和类型建议支持快速选择或搜索式补充

关系区应是“整理入口”，不是“关系跳转导航”。

### 2. 截图资产区

负责：

- 查看截图
- 批量添加截图
- 粘贴截图
- URL 导入截图
- 删除截图
- 预览截图

设计要求：

- 保持明显的网格式资产展示
- 首屏可以有“添加截图”动作，但详细管理放在这个区块
- 对空状态要提供明确的下一步说明，而不是只显示空白

### 3. 评论与备注区

负责：

- 评论
- 整理备注
- 待办信息

设计要求：

- 不再只展示“暂无评论”只读卡片
- 默认进入时就可以开始填写
- 可以保留手动保存，也可以后续演进为自动保存，但当前交互必须直接可输入

### 4. 文件与来源区

负责：

- 网盘文件路径管理
- 来源 URL
- 来源站点
- 与文件相关的后续整理说明

设计要求：

- 文件列表清晰展示名称、路径、体积等关键信息
- 当前页内直接添加和移除文件
- 来源字段与文件管理放在同一区块，避免“来源在上面、文件在下面”造成来回跳读

## 页面角色变化

本次设计的核心不是简单重排，而是改变页面角色：

### 从：展示型详情页

特点：

- 先展示封面、统计、关系摘要
- 真正可编辑动作需要再点“编辑”
- 很多区块只是“告诉用户现在为空”

### 到：整理型工作台

特点：

- 用户进入页面后立即看到下一步能做什么
- 首屏即可完成最重要的整理动作
- 中段继续深挖媒体、关系、文件、备注整理
- 摘要数据只保留辅助价值，不再主导页面结构

## 现有能力复用策略

当前代码中已有以下可复用方向：

- `MovieMetadataSection`：可作为基础信息与来源信息编辑的底稿
- `MovieOrganizationSection`：可作为标签、类型、演员、文件整理区的底稿
- `MovieAssetsSection`：可作为封面、截图管理区的底稿

本期不建议推翻重写，而应做以下调整：

1. 主详情页改成真正组合这些能力。
2. 把“编辑态”和“浏览态”的边界重新定义，减少无意义切换。
3. 让这些区块在视觉层级上形成一个统一工作台，而不是几个隐藏能力组件。

## 交互原则

### 1. 首屏必须可整理

首屏不能再只是：

- 封面占位
- 数量统计
- 空关系摘要
- 只读评论

它必须直接回答：

- 这部片现在处于什么状态？
- 我下一步先补什么？
- 我能立刻在哪改？

### 2. 编辑操作尽量就地完成

尽量避免：

- 点“编辑”之后整页结构大幅跳变
- 某个关系只能跳到别页管理
- 某个媒体动作只能靠隐藏 hover 或二级入口触发

### 3. 摘要数字降级为辅助信息

演员数、标签数、类型数、文件数可以保留，但只作为辅助，不应占据主视觉中心。

### 4. 空状态必须带行动指引

例如：

- 没有封面：提示上传、粘贴、刮削
- 没有截图：提示添加截图、支持 URL 或粘贴
- 没有演员：提供搜索并添加演员入口
- 没有评论：直接显示可输入区域，而不是仅显示“暂无评论”

## 新建影片场景优化

对于刚快速创建、只有番号的新影片，详情页应该天然支持“从空到完整”的整理路径。

进入页面后推荐流程：

1. 补标题、状态、评分
2. 加封面和截图
3. 加标签、类型、演员
4. 补评论、备注、来源与文件

因此页面结构要天然支持这个整理顺序，而不是把用户丢进一个空白摘要页里自行摸索。

## 验收标准

1. 用户进入影片详情页后，可以直接修改状态和评分。
2. 用户可以在当前页直接添加/更换封面与截图。
3. 用户可以在当前页直接编辑标签、类型、演员。
4. 用户可以在当前页直接管理文件与来源信息。
5. 评论和备注在当前页可直接录入，不依赖单独编辑态按钮。
6. 新建影片进入详情页时，第一屏能明确承接后续整理工作。
7. 页面主结构不再以摘要统计卡为核心，而以整理动作为核心。

## 风险与取舍

1. 单页工作台会提高首屏信息密度，因此需要更强的区块层级和留白控制。
2. 如果把所有内容都塞进第一屏，会再次失控，因此必须坚持“第一屏解决最高频动作，中段分区完成深度整理”。
3. 如果继续保留旧的“编辑态 / 浏览态”二元结构过重，用户仍会感觉很多动作被藏起来，因此本期应尽量弱化切换成本。

## 建议实施顺序

1. 先重构 `MovieDetailPage` 的布局骨架。
2. 把封面、基础信息、状态、评分、快速关联整合进第一屏。
3. 接入 `MovieAssetsSection`，形成完整媒体资产区。
4. 接入 `MovieOrganizationSection`，形成标签 / 类型 / 演员 / 文件整理区。
5. 重做评论与备注区，使其默认可编辑。
6. 最后收拢摘要统计与空状态文案，确保整页语义统一。
