# 切换数据库 / 用户工作区设计

## 背景

当前应用在启动时会固定打开一个数据库文件，并将它作为全局 `State<Database>` 注入所有 Tauri 命令。图片、截图、演员头像等资源也默认存放在同一个 app data 目录下的 `images/` 中。这个结构适合单库应用，但不适合“多用户切换”的使用方式。

用户期望的是：进入软件时自动进入默认数据库；在软件内点击一个“切换用户”按钮后，选择另一个用户；切换完成后界面重启并回到默认入口，感觉上就像换了一个账号。

## 目标

1. 提供一个明确的“切换用户”入口。
2. 切换时同时切换数据库和图片资源目录。
3. 切换后重载界面，前端状态回到默认用户的上下文。
4. 保持默认用户可直接进入，不增加额外启动门槛。
5. 尽量复用现有导入、导出、清空数据库能力。

## 非目标

1. 不在本期实现网络同步或多设备账号体系。
2. 不实现复杂的权限控制。
3. 不改变现有数据模型，只改变数据归属和入口。

## 方案选择

采用“工作区目录”模式。每个用户对应一个独立工作区目录，目录内包含：

- `movie-note.db`
- `images/`
- `profile.json` 或等价元数据文件

默认工作区会自动创建，命名为 `default`。用户切换后，应用会切换到另一个工作区目录，并在完成后重载 UI。

与“只切数据库文件”相比，这种方式的优势是：数据库、封面、截图、头像都天然隔离，符合“换账号”的直觉。与“完全独立工作区”相比，本方案不额外引入复杂的偏好同步和运行时拆分，范围更可控。

## 核心设计

### 1. 工作区抽象

在后端引入工作区概念：

- 当前激活工作区路径由一个配置文件或注册文件决定。
- 启动时先读取当前激活工作区，再打开该工作区中的 SQLite 文件。
- 图片保存、截图保存、头像保存全部基于当前工作区目录。

工作区目录建议放在应用数据目录下，例如：

- `AppData/MovieNote/workspaces/default/`
- `AppData/MovieNote/workspaces/alice/`
- `AppData/MovieNote/workspaces/bob/`

### 2. 启动与切换流程

启动流程：

1. 读取“当前激活工作区”指针。
2. 若不存在，自动创建 `default` 工作区。
3. 打开该工作区数据库并执行迁移。
4. 前端按当前工作区进入正常主页。

切换流程：

1. 用户点击“切换用户”。
2. 弹出工作区列表，支持选择已有工作区或新建工作区。
3. 确认后写入新的“当前激活工作区”指针。
4. 前端清理本地状态：筛选、最近访问、列表选择、预览状态、查询缓存。
5. 触发应用重载，或通过命令式重新初始化来恢复到默认首页。

### 3. 前端状态隔离

以下状态应按工作区隔离或在切换时清空：

- 影片筛选条件
- 最近访问记录
- 常用对象频率统计
- 列表滚动位置
- 选中项和预览状态

这样切换后，用户不会看到上一个工作区的操作痕迹。

### 4. 入口位置

切换按钮建议放在现有侧栏的底部操作区，与“导出 / 导入 / 清空”并列，形成统一的数据入口组。这样用户能在同一位置理解：

- 导出当前工作区
- 导入到当前工作区
- 清空当前工作区
- 切换当前工作区

## 数据迁移与兼容

### 默认用户

首次升级后，如果当前只有一个旧数据库文件，则将其视为 `default` 工作区并自动接管，不要求用户手动迁移。

### 旧资源兼容

如果历史图片仍然位于旧的 app data `images/` 目录，需要支持一次性迁移或在加载时兼容查找旧路径。建议优先实现一次性迁移，把旧资源复制到 `default` 工作区中，避免后续路径混乱。

### 导入 / 导出

导出应明确导出“当前工作区”的完整数据；导入应导入到当前工作区，不跨工作区写入。这样工作区之间互不影响。

## 错误处理

切换工作区时要覆盖以下失败场景：

- 目标工作区不存在或无法创建
- 数据库文件无法打开
- 数据库迁移失败
- 工作区配置写入失败
- 前端重载失败

处理原则：

1. 失败时保留当前工作区不变。
2. 给出明确错误提示。
3. 不破坏当前数据库连接。
4. 若切换已写入但重载失败，下次启动仍应进入新的激活工作区。

## 组件与接口边界

后端建议新增或调整的职责：

- 工作区发现、创建、列举、切换
- 当前激活工作区的持久化
- 基于工作区路径创建 `Database`
- 基于工作区路径解析图片保存目录

前端建议新增或调整的职责：

- 侧栏“切换用户”按钮
- 工作区选择对话框
- 切换中的 loading / 错误提示
- 切换后的状态清理与重载

## 验收标准

1. 应用启动后默认进入 `default` 工作区。
2. 能在软件内切换到另一个工作区。
3. 切换后数据库、图片、封面、截图都来自新工作区。
4. 切换后界面回到默认入口，且旧筛选、旧选中状态不再保留。
5. 导入、导出、清空仍只作用于当前工作区。

## 风险与取舍

1. 最大风险是路径迁移不完整，导致旧图片显示异常。
2. 第二个风险是切换后前端缓存未彻底清理，出现“串用户”感知。
3. 第三个风险是把工作区做成“只改 UI 不改后端”，那会造成逻辑不一致，因此必须在后端同步实现。

## 建议实施顺序

1. 先落后端工作区配置和数据库打开方式。
2. 再把图片保存路径改成按工作区分目录。
3. 然后增加前端切换入口和重载逻辑。
4. 最后补迁移与异常提示。
