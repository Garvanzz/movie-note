# 番号规范化与渐进式搜索设计

日期：2026-05-25

## 背景

当前项目已经有基础的番号归一逻辑，但能力仍然偏弱：

- 录入时不会严格按统一规则写入数据库。
- 搜索时只有基础的分隔符兼容，不支持完整的渐进式匹配与排序。
- 不同搜索入口没有共享一套统一的番号与名字匹配规则。
- 演员搜索对中文名、日文名、别名的支持不完整。
- 前端搜索交互仍以“输入后再提交”为主，缺少边输入边收敛的候选下拉。

本次设计目标是在不引入过重搜索系统的前提下，建立一套统一、可解释、可测试的规范化与搜索机制。

## 目标

### 业务目标

- 影片番号在入库时统一转换为规范格式。
- 数据库中只保留规范后的番号，不保留原始输入值。
- 所有与影片和演员相关的搜索入口共享同一套输入规范化与排序规则。
- 搜索在输入过程中实时返回候选，不等待回车。
- 番号搜索支持“前缀先展开，继续输入后收敛”。
- 演员搜索支持中文名、日文名、主名、别名的渐进匹配。

### 非目标

- 不引入全文检索引擎或单独搜索服务。
- 不把所有业务实体改造成统一搜索索引文档系统。
- 不保留原始输入番号字段。
- 不在本次设计中重做整个全局搜索 UI，仅替换其匹配与候选逻辑。

## 已确认的番号规范

### 1. 标准有码前缀型

格式：`PREFIX-NUMBER`

示例：

- `IPX-123`
- `SSIS-045`
- `ABW-007`

规则：

- 前缀统一转为大写。
- 前缀与数字之间统一为单个 `-`。
- 数字段保留有效前导零。

### 2. FC2 型

格式：`FC2-PPV-NUMBER`

示例：

- `FC2-PPV-1234567`

规则：

- 统一补全 `PPV` 段。
- 忽略用户输入中的空格、下划线、大小写和缺失分隔符。

### 3. 素人前缀型

格式：`PREFIX-NUMBER`

示例：

- `SIRO-1234`
- `GANA-2290`
- `LUXU-1925`
- `MAAN-901`

规则：

- 与标准有码前缀型保持一致的分隔规则。

### 4. 带尾缀字母型

格式：`PREFIX-NUMBERSUFFIX`

示例：

- `ABP-123A`
- `MIDE-001B`

规则：

- 尾缀字母直接跟在数字后，不增加额外分隔符。
- 前缀统一大写。
- 数字段保留前导零。

### 5. 日期加流水号型

格式：`DATE-SEQ`

示例：

- `051926-001`
- `051626-01`

规则：

- 下划线统一改写为中划线。
- 最终不增加站点前缀。
- 保持日期段与流水号段结构。

### 6. 纯站点数字型

格式：`SITE-NUMBER`

示例：

- `HEYZO-3850`

规则：

- 纯数字编号必须带明确站点前缀后再入库。
- 用户输入如 `heyzo3850`、`HEYZO 3850`、`heyzo-3850`，都规范为同一结果。

## 规范化设计

### 总体原则

所有写入与搜索都通过同一套规范化模块处理，避免页面级规则分叉。

规范化分成两层：

1. 基础清洗
2. 类型识别与格式重写

### 基础清洗

基础清洗负责处理与业务类型无关的噪音：

- 去除首尾空格。
- 全部英文字母转大写。
- 把空格、下划线、多连字符统一整理。
- 去掉明显无意义的分隔噪音。
- 生成“带分隔符版本”和“去分隔符版本”用于后续匹配。

### 类型识别顺序

为避免误判，输入按以下顺序分类：

1. `FC2` 型
2. 纯站点数字型
3. 标准前缀 + 尾缀字母型
4. 标准有码前缀型
5. 素人前缀型
6. 日期加流水号型
7. 无法识别的普通文本番号

该顺序的目的：

- 先识别最特殊、最容易被普通正则吞掉的类型。
- 避免把 `FC2` 或 `HEYZO` 误判成普通前缀番号。
- 避免把日期加流水号错误提取为“系列”。

### 识别失败策略

如果输入无法明确识别为已知类型：

- 仅保留基础清洗结果。
- 不进行自作聪明的强制改写。
- 仍允许写入与搜索，但不会附带类型特化行为。

该策略用于降低错误规范化风险。

## 存储设计

### Movies

现有 `movies.code` 将继续作为主展示与主识别字段，但其含义调整为：

- `movies.code`：规范后的最终番号

同时保留或新增以下衍生字段，用于搜索与排序：

- `movies.code_norm`：去分隔符版本，用于快速精确匹配与兼容搜索
- `movies.code_kind`：番号类型，例如 `standard`、`fc2`、`amateur`、`date_seq`、`site_numeric`、`unknown`
- `movies.code_prefix`：可选，前缀或站点前缀，用于前缀组搜索与排序
- `movies.code_number`：可选，主数字段或数字主体，用于更稳定的渐进匹配与自然排序
- `movies.code_suffix`：可选，尾缀字母
- `movies.code_sort_key`：可选，排序辅助字段

最终是否拆成多个列，以迁移复杂度和 SQLite 查询可维护性为准。推荐至少保留：

- `code`
- `code_norm`
- `code_kind`
- `code_sort_key`

### Actors

演员搜索需要一个统一的“可搜索名字集合”。

设计上不要求改动现有主展示字段，但需要建立可搜索别名数据来源。若当前表结构不足，可采用以下两种方式之一：

1. 新增演员别名表
2. 在现有演员表中增加可搜索别名字段并做结构化拆分

推荐使用演员别名表，因为更利于多别名扩展和权重控制。

## 搜索设计

### 搜索总原则

所有搜索入口共享同一套输入处理流程：

1. 原始输入
2. 轻量预规范化
3. 判断搜索对象
4. 构造候选查询
5. 统一排序
6. 返回下拉候选或结果列表

### 番号搜索行为

#### 输入 `IPX`

- 视为可用的前缀组查询。
- 返回 `IPX-*` 下的候选结果。
- 候选按最近整理时间、自然番号顺序和精确程度共同排序。

#### 输入 `IPX 123` / `IPX123` / `IPX-123`

- 先预规范为统一搜索表达。
- 再对以下匹配层级打分。

排序优先级：

1. 规范番号完全相等
2. 去分隔符后完全相等
3. 同前缀且数字段前缀相等
4. 规范番号前缀命中
5. 规范番号包含命中
6. 标题等其他影片字段命中

这保证：

- `IPX-123` 永远排在 `IPX-1234` 前。
- 用户即使输入空格或少分隔符，结果仍会自动对齐到规范番号。

### 演员搜索行为

演员搜索必须基于“名字候选集合”，而不是单一字段。

候选集合至少包含：

- 主名
- 中文名
- 日文名
- 别名

排序优先级：

1. 主名完全命中
2. 主名前缀命中
3. 主名包含命中
4. 别名完全命中
5. 别名前缀命中
6. 别名包含命中

目标是：

- 主名优先，但别名可查。
- 输入越长，结果越收敛。
- 中日文名字都能稳定命中。

### 其他搜索对象

标签、类型、系列也应共享“边输入边返回候选”的交互节奏，但匹配逻辑可以较轻：

- 优先前缀命中
- 再包含命中
- 不需要番号那样复杂的结构化解析

### 通用下拉建议行为

适用入口：

- 影片列表页主搜索框
- 顶部全局搜索
- 演员搜索入口
- 刮削弹窗搜索输入框
- 后续所有与影片/演员检索有关的搜索框

交互规则：

- 输入为空时，不展示搜索结果列表，只展示默认提示或最近访问。
- 输入达到最小有效长度后开始请求候选。
- 采用短延迟防抖，避免每次按键都立刻请求。
- 支持键盘上下选择候选。
- Enter 触发当前高亮项。
- Esc 收起候选。
- 输入继续变化时，候选实时收敛。

## 应用范围

本次设计要求统一到所有搜索逻辑，而不是只改一个入口。

优先覆盖范围：

- 影片列表页搜索
- 全局搜索中的影片、演员、标签、类型、系列
- 演员搜索
- 刮削弹窗输入

其中：

- 影片与演员使用统一的规范化与排序基础设施
- 刮削弹窗至少复用输入规范化逻辑，使 `ipx123`、`fc2ppv1234567` 这类输入在提交到外站前先转成稳定格式

## 后端职责

后端负责：

- 番号规范化
- 番号类型识别
- 影片搜索排序与候选返回
- 演员搜索排序与候选返回
- 写入时统一规范
- 迁移已有数据到新规范

这样可以保证：

- 不同前端入口共享规则
- 桌面端与未来其他入口不会产生行为漂移
- 测试点集中

## 前端职责

前端负责：

- 统一搜索输入组件的交互状态
- 防抖
- 下拉候选展示
- 键盘交互
- 结果跳转与筛选动作

前端不负责：

- 自行定义番号规则
- 自行做复杂排序推断

前端可以做轻量预处理，但最终规则以后端为准。

## 数据迁移

需要一次性迁移现有影片数据。

迁移步骤建议为：

1. 为 `movies` 增加新列或刷新现有搜索辅助列。
2. 扫描所有已有 `movies.code`。
3. 对每条记录执行统一规范化。
4. 回填 `code`、`code_norm`、`code_kind`、`code_sort_key` 等字段。
5. 若迁移后出现编号冲突，记录并阻止静默覆盖。
6. 为新字段建立必要索引。

### 冲突策略

如果两个旧值在规范化后变成同一个番号，例如：

- `IPX123`
- `IPX-123`

则视为冲突候选。

处理原则：

- 迁移阶段必须显式发现冲突。
- 不允许静默覆盖。
- 若需要，迁移返回错误并提示人工处理或提供后续冲突解决机制。

## 错误处理

### 录入时

- 若输入可识别，则直接规范化后写入。
- 若输入不可识别但非空，则允许按基础清洗结果保存，避免数据丢失。
- 若规范化后与已有影片番号冲突，则返回明确错误。

### 搜索时

- 搜索输入始终允许模糊、不完整和非规范格式。
- 搜索失败不应报错，只返回空结果。
- 候选接口应保持轻量、快速、幂等。

## 风险与取舍

### 风险 1：错误规范化

如果识别器过于激进，会把用户输入改错。

应对：

- 未识别时只做基础清洗，不强制套格式。
- 先特化识别，再通用识别。

### 风险 2：迁移冲突

旧库中可能已经存在等价但格式不同的重复番号。

应对：

- 迁移阶段显式检测并停止静默覆盖。
- 先保证数据安全，再决定冲突处理 UX。

### 风险 3：搜索体验不一致

如果仍有入口绕过统一后端逻辑，用户体验会继续分裂。

应对：

- 明确本次改造范围是所有搜索入口。
- 前端只保留交互层，规则以后端返回结果为准。

## 推荐实施顺序

1. 抽离并重写后端番号规范化模块。
2. 扩展 `movies` 搜索辅助字段与迁移逻辑。
3. 建立统一的影片候选搜索接口。
4. 建立统一的演员候选搜索接口与别名支持。
5. 接入影片列表页与全局搜索。
6. 接入演员搜索与刮削弹窗。
7. 补齐测试与人工验证。

## 验收标准

满足以下条件视为完成：

- 录入 `ipx123` 最终入库为 `IPX-123`。
- 录入 `fc2ppv1234567` 最终入库为 `FC2-PPV-1234567`。
- 输入 `IPX` 时，影片搜索下拉可自动出现 `IPX-*` 候选。
- 输入继续到 `IPX 123` 时，`IPX-123` 排在第一。
- 演员可通过中文名、日文名、别名搜索到同一条记录。
- 影片列表搜索、全局搜索、演员搜索、刮削弹窗都共享统一的输入与排序逻辑。
- 旧数据迁移后，数据库中影片番号全部符合新规范。
- 测试与构建验证通过。
